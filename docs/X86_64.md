# x86_64 no projeto kernel

Este documento descreve **somente a fração 64 bits** do repositório, com base no estado atual dos arquivos e dos commits mais recentes.

## Objetivo da trilha x86_64

A trilha x86_64 serve para o bring-up do kernel em **long mode**, mantendo o projeto alinhado com a estrutura principal já existente em x86 32-bit.

Hoje ela já permite:

- boot de um kernel 64-bit
- link com script específico ELF64
- instalação de uma IDT 64-bit
- registro de ISRs e IRQs em 64-bit
- remapeamento do PIC em 64-bit
- uso de módulos genéricos do kernel em uma build expandida

## Arquivos centrais

## Boot e entrada

### `boot/boot64.s`
Arquivo de bootstrap do caminho 64-bit.

Função no projeto:

- declara o cabeçalho Multiboot2
- define `_start`
- prepara uma stack inicial
- chama `kernel_main64`

Este é o primeiro ponto de entrada do binário 64-bit.

### `boot/gdt64.s`
Define a GDT usada pela trilha 64-bit.

Há descritores para:

- null descriptor
- código 32-bit
- dados 32-bit
- código 64-bit
- dados 64-bit

O arquivo também expõe `gdt64_descriptor`.

## Linkedição

### `linker64.ld`
Script de link para o binário 64-bit.

Características principais:

- `OUTPUT_FORMAT(elf64-x86-64)`
- `ENTRY(_start)`
- base do kernel em `0x100000`
- separação das seções `.text`, `.rodata`, `.data` e `.bss`

## IDT 64-bit

### `arch/x86_64/idt.c`
Implementa a IDT da trilha x86_64.

Responsabilidades:

- definir a estrutura de entrada da IDT 64-bit
- manter a tabela com 256 entradas
- configurar o descritor da IDT
- executar `lidt`
- instalar ISRs
- instalar IRQs

Funções expostas:

- `idt_init()`
- `idt_install_isrs()`
- `idt_install_irqs()`

### `include/arch/x86_64/idt.h`
Cabeçalho público da IDT 64-bit.

## Exceções em 64-bit

### `boot/isr64.s`
Contém os stubs das interrupções de exceção em 64-bit.

Características:

- define handlers para ISRs 0 a 31
- define ISR 128
- trata exceções com e sem código de erro
- salva registradores gerais
- chama `isr_handler_c`
- retorna com `iretq`

A presença da ISR 128 é relevante para futuras extensões de syscall ou software interrupt handling.

## IRQs em 64-bit

### `boot/irq64.s`
Contém os stubs das IRQs 0 a 15.

Características:

- empilha vetor de interrupção correspondente
- salva contexto de registradores
- chama `irq_handler_c`
- retorna com `iretq`

### `arch/x86_64/irq.c`
Camada C para tratamento das IRQs em x86_64.

Funções e papéis visíveis no estado atual:

- inicialização do subsistema de IRQs
- configuração do temporizador
- contagem de ticks
- base para contagem de segundos
- integração com o scheduler
- tratamento de teclado
- comunicação com o PIC

API pública em `include/arch/x86_64/irq.h`:

- `irq_init(uint32_t timer_hz, uint32_t scheduler_quantum_ticks)`
- `irq_timer_ticks()`
- `irq_timer_seconds()`
- `irq_timer_hz()`
- `irq_handler_c(registers_t *regs)`

## PIC em 64-bit

### `arch/x86_64/pic.c`
Implementa o controle do PIC 8259 na trilha 64-bit.

Funções:

- `pic_remap(int offset1, int offset2)`
- `pic_send_eoi(uint8_t irq)`
- `pic_mask_all()`
- `pic_unmask_irq(uint8_t irq)`

Cabeçalho correspondente:

- `include/arch/x86_64/pic.h`

## Kernel 64-bit

### `kernel/x86_64/kernel.c`
É o núcleo usado no alvo 64-bit expandido.

No estado atual, ele reaproveita a base do kernel e expõe:

- `kernel_main64()` chamando `kernel_main()`

Isso mostra que a trilha 64-bit completa está sendo conduzida para reutilizar componentes do kernel já existentes, em vez de manter um segundo kernel totalmente separado.

## Compatibilidade temporária de VMM

### `kernel/vmm64_stub.c`
Arquivo crucial para entender o estágio atual da fração 64 bits.

Ele implementa uma camada de compatibilidade temporária porque o VMM original ainda é dependente de detalhes 32-bit.

Funções presentes:

- `vmm_init()`
- `vmm_is_enabled()`
- `vmm_wp_is_enabled()`
- `vmm_map_page(...)`
- `vmm_unmap_page(...)`
- `vmm_is_mapped(...)`
- `vmm_translate(...)`

No estado atual, essas implementações são vazias, retornam zero, ou retornam identidade no caso de tradução.

Isso significa que:

- o caminho 64-bit já consegue integrar vários módulos
- o VMM 64-bit real ainda **não foi portado de forma definitiva**

## Funções de apoio reutilizadas na build 64-bit completa

No alvo `kernel64_full.bin`, também entram módulos compartilhados do kernel:

- `kernel/vga.c`
- `kernel/klog.c`
- `kernel/panic.c`
- `kernel/shell.c`
- `kernel/pmm.c`
- `kernel/kmalloc.c`
- `kernel/sched.c`
- `kernel/vfs.c`
- `kernel/ramfs.c`

Há inclusive suporte adicional em VGA para saída hexadecimal 64-bit por meio de `vga_puthex64(uint64_t value)`.

## Alvos de build 64-bit

### `kernel64.bin`
Build mínimo do caminho x86_64.

Ele compila apenas a espinha dorsal do boot e da inicialização 64-bit.

### `kernel64_full.bin`
Build expandido da trilha 64-bit.

Esse alvo reúne:

- kernel 64-bit principal
- drivers e subsistemas compartilhados
- IDT/PIC/IRQ 64-bit
- stubs de interrupção 64-bit

É o alvo mais representativo da fração 64 bits neste momento.

### `check64`
Checagem de compilação freestanding dos arquivos principais do caminho 64-bit.

Útil para:

- confirmar compatibilidade de headers
- detectar regressões em portabilidade 64-bit
- validar integrações antes do link final

## Execução

Fluxo típico:

```bash
make kernel64.bin
make kernel64_full.bin
make iso64
make run64
make check64
```

`make run64` usa `qemu-system-x86_64`.

## Relação com o shell

A build compartilhada incorpora comandos já existentes no shell, inclusive:

- `arch`
- `shutdown`

O comando `arch` verifica se a compilação em execução está em 32 ou 64 bits usando `sizeof(void*)`.

## Leitura correta do estágio atual

A fração 64 bits do projeto já contém:

- entrada própria
- linker próprio
- IDT própria
- PIC próprio
- IRQ própria
- stubs próprios
- alvo de build expandido

Ela ainda **não equivale a um kernel 64-bit totalmente concluído**, sobretudo por causa do VMM temporário.

Portanto, o estágio atual é melhor descrito como:

**bring-up 64-bit funcional e em expansão**, com caminho de interrupções e build já consolidados, mas com subsistemas ainda em adaptação para long mode.

## Próximos passos naturais da trilha x86_64

- portar o VMM real para 64-bit
- revisar a transição efetiva para long mode no bootstrap
- ampliar testes do `kernel64_full.bin`
- unificar gradualmente a organização entre a base x86 e x86_64
